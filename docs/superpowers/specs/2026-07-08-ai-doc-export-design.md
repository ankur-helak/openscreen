# AI Doc Export тАФ Design Spec

- **Date:** 2026-07-08
- **Status:** Approved for planning (pending final spec review)
- **Branch:** `feat/ai-doc-export` (off `feat/caption-model-whisper-base-en`, which carries the
  full AI-voiceover + linked-captions + AI-script-polish foundation, and the export/`FrameRenderer`
  pipeline)
- **Feature area:** new `src/lib/docExport/`, `electron/native-bridge/services/`, `src/native/`,
  `src/hooks/`, `src/components/video-editor/` (export panel/menu), a new `docExport` i18n
  namespace; a small shared-key refactor of the shipped `scriptPolish` seam

## 1. Summary

Add a one-click **AI Doc Export** that turns a recording into a polished, **Trupeer-style visual
product walkthrough** and saves it as a **self-contained HTML file** (plus **PDF**). The document
has a title, an **Overview**, **Who This Guide Is For**, **What You'll Learn**, and **Step-by-Step
Instructions** тАФ where each step is a heading, click-guide prose (real on-screen UI elements
**bolded**), and a **composited screenshot** (cursor / zoom / annotations baked in) captured at that
step's interaction moment.

Steps are **interaction-anchored**: they derive from on-screen events (clicks + zoom/annotation
regions), so screenshots land on the actual button-click, exactly like the reference examples. The
document is generated by a **multimodal** OpenAI call тАФ the model sees both the transcript (for
intent) and each step's screenshot (for the UI detail it describes). Screenshots are
**base64-inlined** into the HTML so images can never break, and the PDF is produced from that same
HTML.

This is the "doc export" feature that the AI Script Polish spec
(`2026-07-07-ai-script-polish-design.md` ┬з1) named as a future consumer of the BYO-key foundation.
It reuses the `scriptPolish` native-bridge seam wholesale for plumbing (BYO OpenAI key in the main
process, single IPC channel, save-via-dialog) and adds three genuinely new pieces: **step
derivation**, **composited-screenshot capture**, and **document assembly + PDF/output**.

## 2. Motivation

Screen recordings are the fastest way to *show* a process, but written documentation is what people
search, skim, translate, and paste into a wiki. Producing that doc by hand тАФ watching the recording,
transcribing steps, screenshotting each moment, writing prose тАФ is exactly the tedious work users
want to avoid. A recording already contains everything a good walkthrough needs: narration (intent),
clicks/zooms (structure and the moments that matter), and composited frames (polished screenshots).
Doc Export assembles those into a finished document in one click.

The infrastructure is already in place: the `scriptPolish` seam proved the BYO-key main-process
pattern, the export pipeline can render composited frames (`FrameRenderer`), and the recording
carries click timestamps + positions, zoom regions, annotation regions, and a transcript. Doc
Export is the highest-leverage way to turn that existing signal into a new, tangible artifact.

## 3. Locked decisions

| # | Decision | Choice | Rationale |
|---|---|---|---|
| 1 | Content generation | **Multimodal тАФ vision + transcript** (`gpt-4o-mini`, which supports images) | Prose must "explain the page being looked at" and name real UI elements; the model must *see* the screen. Transcript supplies intent. |
| 2 | Step model | **Interaction-anchored** тАФ steps derive from clicks + zoom/annotation regions, coalesced; screenshot captured at the interaction instant | Screenshots land on the actual action; step count is deterministic; keeps the AI contract a clean 1:1 (write prose per pre-defined step) reusing the polish id-guard |
| 3 | Step fallback | If a recording has **no click data**, fall back to **narration segments** as step anchors | Robust across platforms/recordings that lack cursor telemetry |
| 4 | Output format | **Self-contained HTML** (screenshots base64-inlined) **+ PDF** (via `printToPDF`) | Inlined images can never break (the failure mode in the reference examples); PDF falls out of the same HTML for near-free |
| 5 | Gating | **Require a transcript** (narrated recording), like Script Polish | Narration materially improves prose; smallest robust v1; interactions enhance when present |
| 6 | Key handling | **Reuse the existing OpenAI key** (BYO-key, `safeStorage`, main process) via a shared `OpenAiKeyStore` | If a key was added for Polish, Doc Export just works; single network + consent choke point; key never enters the renderer |
| 7 | UI placement | **Export panel/menu**, alongside Export MP4/GIF | Doc Export produces an output artifact (a document), not a script edit тАФ it belongs with the other exports, not the Voiceover panel |
| 8 | AI call shape | **One batched multimodal request** returning the whole doc as structured JSON; validated atomically | Coherent doc, shared context, one round-trip; id-set-exact guard gives the same all-or-nothing guarantee as polish |
| 9 | Screenshot source | **Composited frames** from `FrameRenderer`, captured in a **single linear decode pass** | Screenshots include cursor/zoom/annotations (polished look); a linear pass avoids fragile random-access seeking |

## 4. Goals / Non-goals

### Goals (v1)
- One-click **"Export doc"** that produces a self-contained HTML walkthrough + a PDF.
- Document structure: **title тЖТ Overview тЖТ Who This Guide Is For тЖТ What You'll Learn тЖТ
  Step-by-Step Instructions**, each step = heading + click-guide prose + a composited screenshot.
- **Interaction-anchored steps** (clicks + zoom/annotation regions, coalesced), with a
  **narration-segment fallback** when no click data exists.
- **Composited screenshots** captured at each step's interaction moment, **base64-inlined** so they
  never break; downscaled to control file size + token cost.
- **Multimodal generation** via the existing BYO-key OpenAI path, reusing the `scriptPolish`
  main-process seam; **shared OpenAI key** across Polish and Doc Export.
- A **privacy consent/disclosure** that screenshots (the only non-text data) leave the device.
- **Save via the OS dialog**; PDF generated from the HTML.
- Full **i18n parity** for all new strings (13 locales).

### Non-goals (future phases)
- **Markdown output** (and other formats). v1 is HTML + PDF only.
- **Silent-recording support** (generate from clicks + vision with no transcript). v1 requires a
  transcript.
- **Per-step cropping / zoom-to-click** using the click `cx,cy`, callout arrows/highlights drawn
  onto screenshots, or blurring of sensitive regions beyond what the recording already composites.
- **User-editable doc** (review/tweak headings and prose) before export тАФ v1 generates and saves.
- **From-scratch docs** with no recording, or docs from B-roll without narration.
- **Non-OpenAI providers** / local-LLM fallback for generation.
- Branding/logo insertion (the reference examples show a vendor logo; out of scope).

## 6. Module layout (one concern per file)

### 6.1 New тАФ renderer pure libs (`src/lib/docExport/`), unit-testable
- **`types.ts`** тАФ `DocStep`, `DocStepInput` (`{ id, transcriptText }` + image), `GeneratedDoc`
  (`{ title, overview, audience[], learn[], steps:[{id,heading,body}] }`), `DocExportInputs`.
- **`steps.ts`** тАФ `deriveSteps(inputs): DocStep[]`. Collect interaction moments (click `timeMs`
  from cursor samples where `interactionType === "click"`; `ZoomRegion.startMs`;
  `AnnotationRegion.startMs`, excluding `annotationSource === "auto-caption"`), sort, **coalesce**
  moments within `COALESCE_MS` into one step, attach the transcript window spanning each step, and
  choose `screenshotMs` (the representative interaction instant). **Fallback:** when there are no
  clicks, use the narration segments (`voiceover.segments` if present, else
  `segmentTranscript(transcript.segments)`) as step anchors with `screenshotMs = sourceStartMs`.
- **`validateDoc.ts`** тАФ `validateGeneratedDoc(requestedIds, raw): GeneratedDoc`. Reuses the
  id-set-exact pattern from `src/lib/script/validatePolishResults.ts`: throws unless `steps` has
  exactly one entry per requested id (no missing/extra/duplicate) and required doc-level fields are
  present and non-empty; returns steps reordered by `requestedIds`.
- **`renderHtml.ts`** тАФ pure `renderDocHtml(doc, screenshotsById): string`. Builds a self-contained
  HTML document with an inline `<style>` (system-font, callout box, step headings matching the
  reference look) and screenshots as inline `data:image/png;base64,тАж`. Escapes all model text; a
  minimal safe inline-markdown pass renders `**bold**` for UI elements.
- **`index.ts`** тАФ barrel.

### 6.2 New тАФ renderer impure
- **`src/lib/docExport/screenshots.ts`** тАФ `captureStepScreenshots(source, config, stepTimesMs):
  Map<id, Blob>`. Reuses the exporter's decoder + `FrameRenderer` setup (`videoExporter.ts:242`
  `new FrameRenderer(...)`, `:279` `initialize()`, `:376` `renderFrame(videoFrame, tsUs, webcam)`,
  `:378` `getCanvas()`). **Single linear decode pass** startтЖТend: when the decoded frame's timestamp
  crosses the next target, render it, read back the canvas, downscale to `SCREENSHOT_MAX_W`
  (~1280тАУ1536px), encode PNG, advance. Returns one image per step.
- **`src/hooks/useDocExport.ts`** тАФ orchestration hook `{ config?, transcript, тАж } тЖТ
  { status, hasKey, refreshKeyStatus, exportDoc }`. Runs gather тЖТ `deriveSteps` тЖТ capture тЖТ
  `docExport.generate` тЖТ `renderDocHtml` тЖТ `docExport.save`. Status union:
  `{ state: "idle" | "capturing" | "generating" | "rendering" | "saving" | "error"; message? }`.
  On any failure the whole run aborts and nothing is written (atomic).

### 6.3 New тАФ main process
- **`electron/native-bridge/services/openAiKeyStore.ts`** тАФ extract the key logic
  (`readKey`/`getKeyStatus`/`setKey`/`clearKey`, `safeStorage`, DI for tests) currently inside
  `scriptPolishService.ts` into a shared module both services use. Points at a shared
  `userData/openai/openai-key.enc`, with a **one-time migration** from the old
  `userData/script-polish/openai-key.enc` location.
- **`electron/native-bridge/services/docExportService.ts`** тАФ `DocExportService` (constructor DI:
  `configDir`, `fetchImpl`, key store). `generate(steps, context)`: early-return `no-key`; build one
  multimodal request (system prompt with the exact structure + rules; per-step user content =
  `{ id, transcriptText }` + `image_url` with the base64 data URI, `detail: "high"`; plus
  full-transcript context); POST with `Authorization: Bearer`, `response_format:
  { type: "json_object" }`, `AbortSignal.timeout(...)`; map errors to `no-key` / `api-error` /
  `invalid-response`; parse + return `{ success, doc }`. `save(html, formats, ...)`: see ┬з8.

### 6.4 Changed тАФ shipped `scriptPolish` seam (small refactor)
- `scriptPolishService.ts` switches its key handling to the shared `OpenAiKeyStore` (behavior
  unchanged; existing tests updated). This is the "improve the code you're working in" refactor that

### 6.5 Contracts / client / transport

## 7. The generation flow

### 7.1 Inputs assembled by the renderer
Transcript (`useTranscript` тЖТ `Transcript.segments` = `CaptionSegment[]`, `startSec`/`endSec`);
click samples (`interactionType === "click"` with `timeMs`, `cx`, `cy`); `ZoomRegion[]`;
`AnnotationRegion[]` (excluding auto-captions); and `voiceover.segments` when present (preferred
text for the transcript window, since it may be polished).

### 7.2 Step derivation (`deriveSteps`)
Interaction moments are merged into a small number of steps (`COALESCE_MS`, tunable). Each step
carries the transcript text overlapping its span and a single `screenshotMs`. The step **ids** are
stable (`step-1тАжstep-n`) and are the contract with the model.

### 7.3 Screenshot capture (`captureStepScreenshots`)
One linear decode pass produces one composited PNG per step at its `screenshotMs`, downscaled.
Screenshots are held in the renderer and inlined at assembly time; they are sent to the model as
base64 for generation.

### 7.4 The multimodal request (batched, one round-trip)
System prompt fixes the output schema and the rules ("bold real on-screen UI elements", "one step
object per provided id", "describe navigation and what's on screen"). User content interleaves each
step's `{ id, transcriptText }` with its image, plus the full transcript for title/overview.
`response_format: json_object`, model `gpt-4o-mini`.

### 7.5 Response (structured, validated, atomic)
Model returns `{ title, overview, audience[], learn[], steps:[{ id, heading, body }] }`.
`validateGeneratedDoc` enforces exact step-id set + required fields. **Any mismatch тЖТ failed run,
nothing written.**

### 7.6 Assembly + save
`renderDocHtml` interleaves, per step, `heading тЖТ body тЖТ screenshot` (matching the examples), under
the title/overview/audience/learn header, into a self-contained HTML string. `save` writes the
`.html` and, for PDF, writes a temp HTML file, loads it into an offscreen
`BrowserWindow({ show: false })`, calls `webContents.printToPDF`, writes the `.pdf`, and cleans up.
Destination chosen via `dialog.showSaveDialog`.

## 8. Output & save details

- **HTML:** one file, no external assets; every screenshot is `data:image/png;base64,тАж`. Inline
  `<style>` reproduces the reference look (H1 title, overview paragraph, callout box, bulleted
  "Who/Learn", "Step-by-Step Instructions", per-step heading + prose + image). This directly
  prevents the broken-image failure mode.
- **PDF:** produced from the identical HTML via an offscreen window + `printToPDF` (no precedent in
  the repo, but offscreen `BrowserWindow({ show: false })` is an established pattern in
  `electron/windows.ts`). Reuses the `dialog.showSaveDialog` pattern from
  `electron/ipc/handlers.ts`.
- **Save UX:** one dialog; both `.html` and `.pdf` written (final format selection UX тАФ single
  dialog with a format choice vs. both-at-once тАФ settled during planning).

## 9. BYO-key & privacy

- **Storage:** OpenAI key is a **global app setting**, read/written in the main process via
  `safeStorage`, through the shared `OpenAiKeyStore`. Reuses the existing key тАФ **no second key
  UI**; `OpenAiKeyDialog` is shared.
- **Migration:** on first run of the shared store, migrate any key at the old
  `userData/script-polish/openai-key.enc` to `userData/openai/openai-key.enc`.
- **Consent:** screenshots are the only non-text data leaving the device. A clear disclosure in the
  Doc Export UI (and/or a first-use gate in the main process before the first Doc Export network
  call) states that the transcript **and screenshots** are sent to OpenAI. Mirrors the polish
  consent-gate precedent (see `2026-07-07-ai-script-polish-design.md` ┬з9 and the accepted v1
  deviation).
- **Offline default:** with no key set, the rest of the app is unaffected; Doc Export is simply
  disabled with an "Add OpenAI key" affordance.

## 10. Error handling & edge cases

- **No key** тЖТ Doc Export disabled + "Add OpenAI key".
- **No transcript** тЖТ Doc Export unavailable (Decision 5).
- **No clicks** тЖТ step fallback to narration segments (Decision 3).
- **No interactions and no usable narration segments** тЖТ surfaced as "not enough to build a doc"
  (nothing written).
- **Screenshot capture failure** (decode error, blank canvas) тЖТ run aborts with an error status;
  nothing written.
- **API / network / rate-limit / timeout** тЖТ `api-error`; a toast via the existing warnings channel;
  offer retry. Nothing written.
- **Response id mismatch / bad JSON** тЖТ `invalid-response`; failed run; nothing written (atomic).
- **Very long recording** (many steps/screenshots) тЖТ cap step count (coalescing + a max-N with a
  logged notice so truncation is never silent); keep token/image count bounded.
- **Save cancelled** in the dialog тЖТ no-op, back to idle.

## 14. Open questions (non-blocking)
